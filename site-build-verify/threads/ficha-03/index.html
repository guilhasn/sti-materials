    Ficha Pratica 3 - Chat com 2 Clientes (Threads)¶

    Objectivos¶

    • [ ] Perceber claramente a diferenca entre processo e thread.
    • [ ] Implementar um servidor capaz de atender mais do que um cliente em paralelo.
    • [ ] Garantir sincronizacao correta quando ha dados partilhados.
    • [ ] Evitar erros tipicos: race condition, deadlock e bloqueios indefinidos.
    • [ ] Testar o chat com 2 clientes de forma deterministica.

    Conceitos fundamentais (antes de codar)¶

    1) Processo vs Thread¶

    Conceito O que e O que partilha
    Processo Programa em execucao, com espaco de memoria proprio Nao partilha memoria com outros processos por defeito
    Thread Linha de execucao dentro de um processo Partilha memoria, variaveis globais e recursos do processo

    Ideia-chave

    Num servidor de chat, varias threads no mesmo processo permitem atender clientes em simultaneo.

    2) Ciclo de vida de uma thread¶

    • New: criada mas ainda nao iniciada.
    • Runnable/Running: pronta ou a executar no CPU.
    • Blocked/Waiting: a espera de I/O, lock ou sinal.
    • Terminated: terminou com sucesso ou por erro.

    3) Concorrencia e paralelismo¶

    • Concorrencia: varias tarefas progridem no mesmo intervalo de tempo.
    • Paralelismo: varias tarefas executam literalmente ao mesmo tempo (multiplos cores).

    Em sala, o importante e desenhar o codigo para concorrencia correta; o paralelismo real vem do SO/CPU.

    4) Memoria partilhada e secao critica¶

    Se duas threads escrevem na mesma estrutura (ex.: lista de clientes), podes ter corrupcao de estado.

    • Secao critica = bloco que so uma thread pode executar de cada vez.
    • Em C#, usar lock no objeto de sincronizacao.
    private static readonly object clientsLock = new();

lock (clientsLock)
{
    clients.Add(client);
}

    5) Race condition¶

    Acontece quando o resultado depende da ordem de execucao das threads.

    Exemplo: Thread A remove cliente, Thread B envia mensagem para esse cliente no mesmo instante.

    6) Deadlock e Starvation¶

    • Deadlock: duas threads ficam presas a espera uma da outra.
    • Starvation: uma thread fica demasiado tempo sem acesso ao recurso.

    Regra pratica

    Usa poucos locks, em ordem consistente, e mantem o tempo dentro do lock o mais curto possivel.

    7) Conceitos de rede necessarios¶

    • TCP: protocolo orientado a ligacao, fiavel, ideal para chat da ficha.
    • Socket: endpoint de comunicacao.
    • Listener: socket do servidor que aceita novas ligacoes.
    • Stream: canal de leitura/escrita entre cliente e servidor.
    • Protocolo de mensagem: regra para saber onde cada mensagem comeca/termina (ex.: linha por linha com \n).

    Arquitectura recomendada para a Ficha 3¶

    1. Thread principal inicia TcpListener.
    2. Loop de AcceptTcpClient para novas ligacoes.
    3. Para cada cliente aceite, criar uma thread (ou task) dedicada a leitura.
    4. Quando um cliente envia mensagem, servidor faz broadcast para os outros.
    5. Lista de clientes ativa deve ser protegida por lock.
    6. Ao desconectar, remover cliente com seguranca e libertar recursos.
    Servidor (Thread principal)
  -> aceita Cliente A -> Thread Cliente A
  -> aceita Cliente B -> Thread Cliente B

    Passo-a-passo sugerido para os alunos¶

    1. Fazer primeiro servidor para 1 cliente e validar rececao/envio.
    2. Adicionar lista de clientes ligada ao servidor.
    3. Introduzir lock na lista antes de adicionar/remover/enviar.
    4. Criar rotina de broadcast robusta (ignorar cliente desconectado).
    5. Tratar excecoes de rede sem terminar o servidor.
    6. Testar com 2 clientes em paralelo e registar comportamento.

    Checklist de testes (obrigatorio)¶

    • [ ] Dois clientes conectam sem bloquear o servidor.
    • [ ] Mensagem do Cliente A chega ao Cliente B.
    • [ ] Mensagem do Cliente B chega ao Cliente A.
    • [ ] Desconexao de um cliente nao derruba o servidor.
    • [ ] Reconexao posterior continua funcional.
    • [ ] Nao existem excecoes nao tratadas no terminal.

    Erros comuns e como evitar¶

    Erro Causa tipica Como corrigir
    Servidor bloqueia no primeiro cliente Leitura de cliente na thread principal Mover leitura para thread dedicada por cliente
    Mensagens "desaparecem" Race condition na lista de clientes Proteger add/remove/iteracao com lock
    Crash ao desconectar Stream/socket nao tratados Usar try/catch/finally e remover cliente no finally
    Broadcast incompleto Protocolo de mensagens mal definido Definir delimitador e parsing consistente

    Boa pratica de avaliacao

    Entregar codigo com logs simples (ligou, enviou, recebeu, saiu). Isso facilita debug e prova funcionamento.

    Criterios de correcao (sugestao)¶

    • Modelo de concorrencia correto (30%)
    • Sincronizacao e seguranca de dados partilhados (25%)
    • Comunicacao cliente-servidor funcional com 2 clientes (25%)
    • Tratamento de erro e encerramento limpo (10%)
    • Clareza de codigo e explicacao tecnica (10%)
